Case study — fluxul comercial fragmentat

Problemă

Echipa schimba 3 suprafețe diferite pentru același flux comercial (lead, follow-up, billing), ceea ce adăuga fricțiune și întârzia actualizările recurente.

Decizie

Am unificat interacțiunile într-un dashboard modular, cu acțiuni contextuale și blocuri RBAC: utilizatorii văd doar pașii relevanți rolului lor.

Rezultat

De la ~7 click-uri la 2 pentru update-uri recurente. Fluxul principal rulează într-un ecran unic, fără salturi între pagini pentru task-uri dependente.

Cum am măsurat: fluxul „actualizare status + reasignare + follow-up”, numărând click-urile în varianta fragmentată vs board-ul unic. Este un baseline de auto-test, nu un studiu cu utilizatori externi.

Decision Log

Decizii tehnice și de produs — cu rațional complet

Fiecare decizie majoră din construirea Alpis Fusion a implicat un trade-off real. Am documentat alternativele considerate, ce am ales și de ce — inclusiv unde aș decide diferit azi.

Arhitectură: single-file HTML vs Vite + React
Ales: Vite + React

Alpis Fusion v1 era un fișier HTML de ~1900 linii. Funcționa, dar orice modificare minoră risca să rupă altceva. Nu existau componente reutilizabile — carduri similare erau copy-paste cu variații.

Alternativă
Continui cu single-file HTML/CSS/JS, mai simplu de deploy, fără build step.
✓ Ales
Migrez la Vite + React: componente reutilizabile, HMR în dev, build optimizat, GitHub Actions CI/CD.
De ce: La ~600 linii de JS inline am ajuns la primul moment de „nu mai știu ce rupe ce”. Separarea componentelor a rezolvat asta structural, nu cosmetic. GitHub Actions a adăugat deploy automat fără FTP. Azi aș face la fel, dar aș migra mai devreme — la ~300 linii, nu la 1900.
🔧 Tehnic📦 Build optimizat🚀 CI/CD
UX: densitate mare vs wizard pas cu pas
Ales: densitate mare

Pentru un CRM operațional există două filosofii opuse: arată tot pe un ecran (densitate) sau ghidează utilizatorul pas cu pas (wizard). Trade-off-ul e real: scanare rapidă vs onboarding mai ușor.

Alternativă
Wizard: un pas pe ecran, progres vizibil, mai puțin copleșitor pentru utilizatori noi.
✓ Ales
Dashboard dens: toate informațiile relevante vizibile simultan, cu priorități susținute de culoare și poziție.
De ce: CRM-urile sunt folosite de operatori experți, nu de vizitatori ocazionali. Scanarea rapidă a 20 de lead-uri e mai valoroasă decât un onboarding lin pentru cineva care deschide dashboard-ul de 10 ori pe zi. Am compensat cu semnale vizuale clare. Trade-off acceptat conștient: curba de onboarding e mai abruptă pentru utilizatori noi.
🎯 UX⚡ Eficiență operațională
State management: Context API vs Zustand vs Redux
Ales: Context API + useReducer

Cu 6 module care partajează stare (lead selectat, filtru activ, status kanban, date factură), state management-ul era o decizie critică de arhitectură.

Alternativă
Zustand sau Redux Toolkit: mai predictibil pentru stare complexă, DevTools mai bune, mai ușor de testat.
✓ Ales
Context API + useReducer: zero dependențe externe, suficient pentru complexitatea actuală, mai simplu de înțeles la revenire.
De ce: La scara actuală, dependența de Zustand sau Redux adăuga complexitate fără beneficiu real. Context API cu un reducer centralizat a acoperit toate nevoile. Azi aș reconsidera dacă proiectul crește: Zustand e mai puțin verbos decât useReducer pentru stare complexă.
🏗️ Arhitectură📦 Bundle size
Autosave: la fiecare keystroke vs buton explicit Save
Trade-off acceptat

Autosave schimbă fundamental comportamentul utilizatorului. Fără buton Save, oamenii testează mai mult și editează mai liber. Dar introduce riscul de salvări accidentale.

Buton Save explicit
Utilizatorul controlează când se salvează. Mai sigur, mai familiar, dar adaugă un click în fiecare interacțiune.
✓ Ales: Autosave
Salvare automată cu debounce de 800ms + indicator vizibil „Salvat” + posibilitate de undo pe 30 secunde.
De ce: A fost decizia de UX cu cel mai mare impact și a costat puțin cod. Autosave + undo e superior butonului Save în context operațional — reduce fricțiunea fără să elimine siguranța. Trade-off: undo rezolvă parțial salvările accidentale, dar nu complet.
🎯 UX⚡ Eficiență⚠️ Risc: edit accidental
Backend real vs date simulate (localStorage)
Respins: backend real

Un CRM real are nevoie de un backend pentru date persistente multi-utilizator. Alegerea a fost între a simula totul în localStorage sau a construi un API.

Respins
Backend real (Node.js + PostgreSQL sau Supabase): date persistente, multi-utilizator, sincronizare în timp real.
✓ Ales
Date simulate complet în JavaScript + localStorage: 0€ hosting, deploy instant pe GitHub Pages, focus pe UI/UX nu pe infra.
De ce: Scopul proiectului este demonstrarea logicii de produs și a abilităților front-end, nu construirea de infra. Un backend real ar fi adăugat 3-4 săptămâni de lucru pentru ceva ce un recruiter nu vede. Limitare conștientă: datele nu persistă între sesiuni diferite.
💰 0€ hosting🚀 Deploy instant⚠️ Limitare: sesiune single-browser
RBAC: rol hardcodat vs selector de rol în UI
Ales: selector de rol în UI

Role-Based Access Control presupune că fiecare utilizator are un rol. Într-un demo fără backend, cum demonstrezi RBAC convingător?

Alternativă
Hardcodez un rol implicit (Admin) și arăt toate permisiunile — mai simplu, dar nu demonstrează RBAC real.
✓ Ales
Selector vizibil în UI: Admin, Manager, Agent. Schimbarea rolului ascunde/arată secțiuni instant, fără reload.
De ce: Un demo de RBAC care nu demonstrează schimbarea de rol nu demonstrează nimic. Selectorul vizibil permite unui recruiter sau evaluator să vadă imediat cum se comportă UI-ul pentru fiecare rol — asta e valoarea reală a feature-ului.

Flux fragmentat → workspace unic

Înainte
  • 3 tab-uri deschise simultan: CRM, task manager, factură
  • ~7 click-uri pentru un update de status + follow-up
  • Date nesincronizate: statusul din CRM nu se reflecta în task
  • Zero automatizare: fiecare tranziție manuală
După
  • 1 ecran: lead, task, invoice în același context
  • ~2 click-uri pentru același update
  • Stare sincronizată: statusul se propagă în toate modulele
  • Flow builder: tranziții automate configurabile
