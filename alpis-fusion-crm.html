1. Contextul și motivația

Alpis a pornit dintr-o întrebare simplă: cum arată un CRM care nu cere training de 2 ore? Majoritatea CRM-urilor pe care le-am întâlnit în 5+ ani de marketing și CRM ops aveau aceeași problemă — utilizatorul trecea între 3-4 suprafețe diferite pentru un singur flux comercial. Lead aici, task acolo, factură în alt tab, calendar într-al patrulea.

Am ales să construiesc nu un clone Salesforce, ci un CRM opinionated: cu o singură cale clară prin fluxul comercial, optimizat pentru cei care vând și operează în paralel.

User story principal

„Ca operator comercial, vreau să închid un flux complet — de la lead la factură — fără să-mi schimb tool-ul sau să pierd context."

Am ignorat conștient 4 user stories care păreau importante (raportare avansată, integrări terțe, mobile-first, multi-tenant). Trade-off: mai bine un produs care rezolvă o problemă bine decât 5 jumătate.

2. Constrângeri reale

  • Solo dev. Eu, weekend hours, ~6-8 ore/săptămână efectiv pe cod.
  • Zero buget hosting. GitHub Pages = static hosting, fără backend, fără DB.
  • Zero buget servicii. Nu folosesc Auth0, Stripe API live, nimic plătit. Toate features simulate frontend-only cu localStorage.
  • Audience target: recruiteri tech și PM-i din SaaS B2B. Nu utilizatori reali — deci UX trebuie să fie credibil ca demo SaaS, nu funcțional ca produs comercial.
  • Bilingv obligatoriu. Toate texte RO + EN, schimbate dinamic.

Aceste constrângeri au format în mod fundamental deciziile tehnice. Fără backend înseamnă state management agresiv în client. Fără DB înseamnă localStorage cu schemă disciplinată. Audience demo înseamnă polish UI ridicat, nu acoperire de edge cases.

3. Decision Log — 8 alegeri majore

Pentru fiecare alegere notez: ce am ales, ce am respins, de ce, și trade-off-ul acceptat.

Decision 01

Vite + React, nu Create React App sau Next.js

Ales: Vite 5 + React 18 + TypeScript 5.

Respins: CRA (abandonat oficial 2023), Next.js (overkill pentru static, plus complexitate App Router).

De ce: dev server pornește în ~50ms vs 5-10s la CRA. Build cu esbuild + Rollup. Plugin ecosystem matur.

+ CâștigDX excelentă, build rapid, comunitate activă, hot reload instant
− CostCurve de învățare puțin diferită vs Webpack tradițional
Decision 02

Single-file prototip până la ~1917 linii

Ales: tot codul într-un singur component până când durerea > beneficiul.

Respins: feature folders din ziua 1, atomic design, Storybook setup early.

De ce: exploratory phase. Refactoring prematur îngheață forme care încă nu sunt corecte. La 200 linii nu știi care va fi structura finală.

+ CâștigIterație ultra-rapidă pe UX, descoperire reală a formei produsului
− CostRefactor mare la final (3-4 zile), risc de bug-uri în transition
Decision 03

localStorage ca DB, nu IndexedDB sau Supabase

Ales: localStorage cu schemă strictă (`alpis:leads:v1`, `alpis:tasks:v1`).

Respins: IndexedDB (overkill pentru <5MB date), Supabase (cont, networking, complexitate).

De ce: demo offline-first, zero networking, instant rendering. Plus versionare în cheie pentru migrații (`v1` → `v2`).

+ CâștigZero latency, zero cost, demo funcțional offline
− CostLimită ~5MB, fără sync între device-uri, fără backup automat
Decision 04

Command Palette ⌘K ca primary navigation

Ales: ⌘K (sau Ctrl+K) deschide command palette cu fuzzy search, acțiuni și navigare rapidă.

Respins: sidebar permanent extins, top nav cu mega-menu.

De ce: demonstrează că target audience (power users SaaS) preferă tastatura. Plus oferă feature memorabil care diferențiază demo-ul.

+ CâștigDiferențiator vizibil, target audience înțelege imediat valoarea
− CostUseri necunoscători nu descoperă feature-ul fără hint vizual
Decision 05

Wizard pentru lead creation, nu form lung

Ales: creation flow în 4 pași cu progres permanent vizibil.

Respins: form clasic cu 12+ câmpuri scrollabile.

De ce: form abandon rate cunoscut scade dramatic la 3-5 pași calibrați. Plus permite validare contextuală pe pas.

+ CâștigCompletion rate ridicat, validare clară, mai puțin overwhelming
− CostMai multe click-uri pentru utilizatori avansați care vor totul deodată
Decision 06

RBAC simplificat — 3 roluri, nu permission matrix complexă

Ales: Owner, Manager, Operator. Permisiuni hardcoded per rol.

Respins: permission matrix granulară (read/write/delete per resource).

De ce: 80% din casurile reale CRM SMB se mapează pe 3 roluri. Permission matrix introduce complexitate UI și mental load fără ROI clar.

+ CâștigUI simplă pentru rol assignment, mental model clar
− CostCazurile edge (custom permissions) cer cod special
Decision 07

Flow Builder vizual, nu YAML config

Ales: drag-and-drop nodes (trigger → condition → action) cu connect lines.

Respins: YAML/JSON config files (Zapier-style API config).

De ce: diferențiator vizibil în demo. Plus target audience non-technical.

+ CâștigWow factor în demo, accesibil non-tech
− CostImplementare canvas/SVG complexă (~3 zile)
Decision 08

Deploy ca prima feature, nu ultima

Ales: GitHub Actions configurat de la commit-ul #1, push pe main → deploy live.

Respins: deploy la final după "lansare", manual rsync, drag-drop la Netlify.

De ce: dacă nu pot vedea ce am construit live, nu construiesc. Plus prinde devreme problemele de build (path-uri, env vars, lock files).

+ CâștigFeedback loop instant, demo permanent updated, încredere la stakeholders
− Cost~4h setup inițial pentru workflow, secrets, permissions

4. Deep-dive tehnic — 3 features

4.1 Command Palette ⌘K

Implementat ca portal React cu fuzzy search peste 4 colecții: navigation, leads, tasks, settings.

// Pseudo-code structure
function useCommandPalette() {
  const [query, setQuery] = useState('');
  const [open, setOpen] = useState(false);
  
  useHotkey('cmd+k', () => setOpen(o => !o));
  
  const items = useMemo(() => fuzzy(query, [
    ...navigationItems,
    ...leadsAsCommands,
    ...tasksAsCommands,
    ...settingsItems
  ]), [query]);
  
  return { open, items, setQuery };
}

Trade-off: fuzzy match folosește algoritm simplu (substring scoring), nu Fuse.js. Bundle: 0KB suplimentar față de ~12KB cu Fuse. Calitatea match-ului: 80% suficientă pentru <500 items.

4.2 Booking Conflict Detection

Pentru calendar booking-uri, detectez conflicte cu interval tree implementat manual:

function detectConflicts(newBooking, existingBookings) {
  return existingBookings.filter(b =>
    b.userId === newBooking.userId &&
    !(newBooking.end <= b.start || newBooking.start >= b.end)
  );
}

Pentru <1000 booking-uri, O(n) e suficient. Pentru scaling, ar trebui interval tree real (O(log n)). Dar acela e overengineering la nivelul actual.

4.3 Flow Builder cu drag-and-drop

Folosesc HTML5 Drag and Drop API native, fără react-dnd. Connections desenate cu SVG paths cubice. State sincronizat cu localStorage la fiecare schimbare.

Probleme rezolvate: drag preview ghost (custom cu cloneNode + setDragImage), drop zones invalide (validare în onDragOver), undo/redo cu stack de snapshots.

5. Probleme reale & cum le-am rezolvat

5.1 GitHub Actions deploy fail — workflow file placement

Prima încercare am pus workflow-ul în workflows/deploy.yml în loc de .github/workflows/deploy.yml. Build a rulat doar pe push, nu pe PR. Pierdere de 2 ore.

Lecția: documentația GitHub e foarte specifică pe path-uri. Read carefully.

5.2 npm ci vs npm install

Workflow-ul rula npm install care actualiza package-lock.json în CI, lock-ul nu era commit-at, build-ul devenea non-reproducibil.

Soluție: commit `package-lock.json`, folosesc `npm ci` în workflow (mai rapid, strict mode).

5.3 Vite base path pentru GitHub Pages

Pe GitHub Pages, app-ul e la /codepen-portfolio/, nu la root. Default Vite build folosește root absolute paths — toate asset-urile dau 404.

Soluție: în vite.config.ts: base: '/codepen-portfolio/'. Plus folosesc import.meta.env.BASE_URL pentru asset-uri dinamice.

5.4 localStorage quota exceeded

La ~50 lead-uri cu attachments base64, am atins limita de 5MB localStorage. Browser arunca QuotaExceededError tăcut.

Soluție: wrapper around localStorage cu try/catch, plus strategie de pruning (oldest-first) când quota se apropie. În prod aș folosi IndexedDB.

7. Ce aș face altfel

  • Aș scrie teste mai devreme. Nu TDD strict, dar 5-10 teste pe paths critice (booking conflict, RBAC enforcement, flow validation) ar fi prins bug-uri pe care le-am descoperit manual la 6 săptămâni.
  • Aș folosi Tailwind de la început. Am pierdut ~3 zile cumulativ pe naming conventions CSS modules. Tailwind ar fi rezolvat 80% din nevoie cu zero gândire.
  • Aș face deploy preview pentru fiecare PR. Vercel sau Netlify Deploy Previews. Reviewing static screenshots e nerealistic.
  • Aș extrage tipurile TypeScript în types.ts din ziua 1. Le aveam împrăștiate, refactor-ul a durat o zi întreagă doar pentru asta.
  • Aș documenta decision log în repo, nu în Notion. Linking-ul e mai natural cu fișiere markdown în docs/ decât cu URL-uri externe care se schimbă.
  • Aș face accessibility audit săptămânal, nu la final. Am avut probleme cu focus management în Command Palette pe care un quick axe scan le-ar fi prins instant.
