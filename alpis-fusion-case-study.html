01 · Context

De ce am construit un CRM de la zero

Background-ul meu în CRM și marketing digital m-a expus la o problemă recurentă: echipele de vânzări și operațiuni foloseau 3–5 tool-uri separate (pipeline, task-uri, facturare, booking, comunicare) care nu se vorbeau între ele. Rezultatul era context switching constant și date fragmentate.

Alpis Fusion CRM Premium nu este un exercițiu de UI. Este un produs de referință construit pentru a demonstra că un front-end developer cu background în produs poate gândi și implementa un workspace integrat — nu doar un dashboard frumos.

„Un CRM bun nu organizează date — organizează decizii. Fiecare ecran trebuie să răspundă la o întrebare clară: ce fac acum cu acest lead / task / client?"

— Principiu de design ghid pentru Alpis Fusion

Stack-ul ales — React 18 + Vite + GitHub Actions — reflectă o decizie conștientă de a folosi unelte industriale reale, nu confort. Am evitat component libraries pentru că voiam control total asupra fiecărui comportament vizual și a fiecărei interacțiuni.

03 · Decision Log

Alegerile de produs documentate

Fiecare decizie majoră a generat un conflict între opțiuni — le documentez cu contextul, alternativele și rațiunea finală.

D-01 · Single workspace vs. multi-page navigation
ARCHITECTURAL
Context: Primele prototipuri aveau navigație separată per modul (pipeline.html, tasks.html, billing.html). Usability testing intern arăta că schimbul de context era principalul blocaj.
Opțiunea respinsă
Multi-page: fiecare modul pe URL propriu
Navigație clasică cu sidebar link-uri
State resetat la fiecare navigație
Context switching: 7+ click-uri per update
Opțiunea aleasă
Single Page App cu tab switching
Toate modulele accesibile fără reload
State persistent între tab-uri
Context switching: 1–2 click-uri per update
Decizie: SPA cu React Router + lazy loading per modul. Trade-off acceptat: bundle inițial mai mare (~287KB), compensat de experiență fluidă și LCP optimizat.
D-02 · Kanban vs. List view pentru pipeline
UX TRADE-OFF
Context: Pipeline-ul de lead-uri trebuia să fie atât vizibil (status rapid) cât și scanabil (volum mare de contacte). Ambele views au avantaje distincte.
Kanban
Vizibil: status pe etape la o privire
Drag & drop intuitiv pentru mutare
Limitat la ~10 carduri/coloană practic
Căutare și filtrare mai greoaie
List / Table
Scanabil: volume mari de contacte
Filtrare și sortare rapidă
Export CSV direct
Nu comunică statusul vizual la fel
Decizie: Ambele. Toggle Kanban/List cu stare salvată per sesiune. Kanban pentru overview operațional, List pentru bulk actions și export. Costul: complexitate UI mai mare, compensat de utilitate.
D-03 · Flow Builder: nod drag-and-drop vs. form sequential
FEATURE DESIGN
Context: Automatizările de follow-up trebuiau să fie definite de user. Drag-and-drop este familiar din tools ca Make/Zapier, dar adaugă complexitate de implementare semnificativă.
Drag-and-drop canvas
Familiar din Zapier / Make
Vizual, flexibil
+2 săptămâni implementare
Acces. problematică pe touch
Step-based sequential
Implementat în 3 zile
Accesibil, clar, linear
Acoperă 90% din use cases reale
Mobile-friendly by default
Decizie: Step-based cu trigger → condition → action pattern. Acoperă use cases reale (email la X zile, task la schimbare status) fără overhead. Drag-and-drop canvas rămâne în roadmap v2.
D-04 · RBAC: roluri fixe vs. roluri custom
PRODUCT INSIGHT
Context: Role-Based Access Control era necesar pentru a simula un CRM real cu echipă. Întrebarea: cât de granular să fie permisiunile?
Roluri custom (granular)
Flexibil, enterprise-ready
Complexitate admin ridicată
Dificil de înțeles pentru SMB
Edge cases de implementare numeroase
3 roluri fixe + toggle
Admin / Manager / Agent — clar
Acoperă 95% din cazuri SMB
Implementat în 1 zi
UX simplu: toggle per secțiune
Insight: Complexitatea nu este o virtute. Roluri fixe cu toggle clar acoperă 95% din nevoile reale ale echipelor mici-medii. Simplitatea este o decizie de produs, nu o limitare tehnică.
D-05 · Billing: PDF invoices native vs. link extern
UX TRADE-OFF
Context: Modulul de billing trebuia să genereze facturi. Alternativa simplă era un link către un tool extern (Stripe, FreshBooks). Alternativa ambițioasă: PDF generat în browser.
Link extern (Stripe etc.)
Implementat în ore
Nu demonstrează capabilitate
Dependency pe third-party
PDF generat în browser
Demonstrează capabilitate tehnică
Zero dependențe externe
Template customizabil
Decizie: PDF generat nativ cu CSS print + window.print() + template HTML structurat. Funcționează offline, zero cost, și demonstrează că front-end-ul poate acoperi fluxuri complete fără backend.

05 · Trade-offs conștiente

Ce am ales să NU fac și de ce

Deciziile de excludere sunt la fel de importante ca cele de includere. Le documentez explicit.

❌ Backend / API propriu
EXCLUS CONȘTIENT
Motivul: Scopul proiectului este demonstrarea capabilităților front-end, nu arhitectura full-stack. Un backend ar fi mutat atenția de la UX și logica de produs la DevOps și securitate. localStorage + JSON export acoperă complet demonstrația. Un backend real ar necesita hosting plătit și ar reduce accesibilitatea demo-ului pentru recruitori.
❌ Component Library (MUI, Chakra, Shadcn)
EXCLUS CONȘTIENT
Motivul: O component library ar fi ascuns deciziile de design în spatele unor defaults predefinite. Scopul era să demonstrez că știu să construiesc interfețe, nu să configurez biblioteci. CSS custom înseamnă că fiecare decizie vizuală este a mea — și poate fi explicată.
❌ TypeScript strict pe tot proiectul
AMÂNAT
Motivul: TypeScript strict crește timpul de implementare cu ~30% pentru un solo developer. Am ales să demonstrez logica de produs completă în JavaScript, cu tipizare pe modulele critice (RBAC, billing). TypeScript complet este în roadmap pentru versiunea enterprise.
