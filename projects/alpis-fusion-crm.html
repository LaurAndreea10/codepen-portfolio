1. Contextul și motivația

Alpis a pornit dintr-o întrebare simplă: cum arată un CRM care nu cere training lung? În multe fluxuri comerciale, utilizatorul trece între lead, task, factură și calendar pentru o singură acțiune completă.

Am ales să construiesc un CRM opinionated: o singură cale clară prin fluxul comercial, optimizată pentru persoane care vând și operează în paralel.

User story principal

„Ca operator comercial, vreau să închid un flux complet — de la lead la factură — fără să-mi schimb tool-ul sau să pierd context.”

Am lăsat intenționat pe plan secund raportarea avansată, integrările terțe și mobile-first. Trade-off: mai bine un produs care rezolvă o problemă bine decât 5 jumătate.

2. Constrângeri reale

  • Solo dev. Iterații în timp limitat, cu focus pe părțile demonstrabile.
  • Zero buget hosting. GitHub Pages, fără server propriu.
  • Zero costuri externe. Date locale, demo funcțional fără servicii plătite.
  • Audience target: recruiteri tech și PM-i din SaaS B2B.
  • Bilingv obligatoriu. Textele RO/EN trebuie să rămână coerente în UI.
Constraints & Trade-offs

Am ales polish UI + flux complet în detrimentul unui backend real. Pentru portofoliu, valoarea este să arăt gândire de produs, arhitectură front-end și interacțiune verificabilă. Persistența reală multi-device rămâne roadmap, nu promisiune ascunsă.

3. Decision Log — alegeri majore

Deciziile sunt documentate ca alegere, alternativă respinsă și cost acceptat.

Decision 01

Vite + React, nu framework complex

De ce: feedback loop rapid, build static simplu și deploy previzibil pe GitHub Pages.

+ Câștig

Iterare rapidă și demo permanent actualizabil.

− Cost

Atenție extra la base path și asset-uri statice.

Decision 02

Prototip dens înainte de refactor

Am păstrat multă logică împreună până când forma produsului a devenit clară. Refactor-ul a fost costisitor, dar a venit după validarea fluxului.

Decision 03

Date locale pentru demo

Am ales date locale ca să păstrez proiectul fără costuri, rapid și ușor de testat. Cost: fără sincronizare între device-uri.

Decision 04

Command Palette ca accelerare

Comanda rapidă este pentru power users: nu înlocuiește navigarea vizibilă, dar scurtează fluxurile repetitive.

Decision 05

Flow Builder vizual

Am ales noduri vizuale în loc de configurare text, pentru că targetul non-tehnic înțelege mai rapid automatizările văzute ca pași.

4. Features analizate

4.1 Command Palette

Permite acces rapid la navigare, lead-uri, task-uri și setări. Trade-off: trebuie semnalizată vizual, altfel utilizatorii noi nu o descoperă.

4.2 Booking conflict detection

La dimensiunea demo-ului, o verificare simplă este mai potrivită decât o structură complexă. Asta păstrează codul ușor de urmărit.

4.3 Flow Builder

Flow Builder-ul arată cum pot fi conectate trigger, condiție și acțiune într-un mod vizual, apropiat de modul în care non-developerii gândesc procesele.

5. Probleme reale & cum le-am rezolvat

5.1 Deploy static și path-uri

Pe GitHub Pages, asset-urile trebuie gândite pentru subfolder, nu pentru root. Am tratat asta ca problemă de build, nu ca problemă de design.

5.2 Refactor după prototip

Prototiparea rapidă a dus la un fișier mare. Soluția a fost separarea logicii pe module după ce produsul a devenit clar.

5.3 Persistență locală limitată

Datele locale sunt excelente pentru demo, dar nu sunt suficiente pentru produs real. Am notat limita explicit în roadmap.

7. Dovadă live vs postere

Transparență

Imaginea OG este poster SVG de prezentare, nu captură reală. Dovada principală este demo-ul live. Următorul pas recomandat: GIF scurt cu fluxul „lead → task → invoice”, ca să nu depindă de iframe.

Stack decisions — motivul, nu doar badge-ul

Vite

Feedback loop rapid și build static simplu.

React

Componente reutilizabile pentru module diferite, dar cu structură comună.

TypeScript

Modele clare pentru lead, task, booking și roluri.

GitHub Actions

Deploy automat ca verificare continuă a build-ului.

8. Ce aș face altfel

  • Aș scrie teste mai devreme. Câteva teste pe paths critice ar fi prins bug-uri descoperite manual.
  • Aș proiecta schema înainte de UI. Unele refactor-uri au venit fiindcă forma datelor a fost dedusă prea târziu din interfață.
  • Aș adăuga un GIF real. O captură scurtă a fluxului ar fi mai puternică decât orice poster.
  • Aș face audit de accesibilitate săptămânal. Focus management-ul trebuie verificat devreme, nu la final.

9. Roadmap public

  • Persistență cross-device: trecerea de la date locale la date sincronizate.
  • Mobile-first redesign: varianta actuală este desktop-optimal.
  • Integrări: webhook-uri și exporturi mai clare.
  • Scoring asistat: sugestii smart pentru lead-uri și task-uri.
